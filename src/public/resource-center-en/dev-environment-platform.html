Environment Details

DEV · SIT · UAT · STAGING · PROD · COB

Six environment types cover the full R&D delivery lifecycle and can be enabled, combined, and extended as needed, such as adding PreSIT, PT performance environments, or DR disaster recovery environments.

DEV

Development Environment

Development

A rapid iteration environment for developers, allowing frequent deployment and debugging.

  • Independent deployment for feature branches with second-level feedback
  • Use unified dependency packages and local subsets of desensitized data
  • Accessible only to internal R&D teams with lightweight approval
Frequent ChangesDeveloper Self-TestingLow Data Sensitivity
SIT

System Integration Test

System Integration Test

A joint debugging environment for multiple services and teams, validating interfaces and integration correctness across systems.

  • API contracts, service orchestration, and dependency connectivity validation
  • Unified desensitized sample data to ensure repeatable testing
  • Automated integration tests executed with CI/CD
Integration ValidationAPI ContractsAutomated Testing
UAT

User Acceptance Test

User Acceptance Test

A production-like environment for business and testing teams to perform acceptance testing, closely matching real usage.

  • End-to-end acceptance of business processes
  • Use standardized desensitized business sample data
  • Changes require business confirmation with medium approval strength
Business AcceptanceProduction-likeControlled Access
STAGING

Staging Environment

Pre-Production

A simulation environment highly consistent with production, used for final validation before release.

  • Production-homogeneous configuration and same-version dependencies
  • Canary releases, regression testing, and release rehearsals
  • Strict approval and change window management
Production SimulationCanary RegressionRelease Rehearsal

Production Environment

Production

The environment that provides official external services, with strict control, auditing, and monitoring.

  • Real data with the highest data security and compliance level
  • Changes require multi-level approval and release tickets
  • Full-link monitoring, alerts, and audit trails

Separation of Duties

Development, testing, release, and operations permissions are separated, with least privilege for critical environments to meet audit and internal control requirements.

▤

Traceable Changes

Every promotion and release has release tickets, approval records, and artifact fingerprints, making every action evidence-backed and auditable.

✓

Compliance Built In

Compliance checks, security scans, and policy gates are built into the pipeline instead of being patched in before launch.

01

Unified Infrastructure Dependency Packages

Use unified base images, dependency manifests, and artifact repositories to eliminate “environment inconsistency,” one of the most common delivery risks.

▣

Unified Base Images and Runtimes

Operating system, runtime, and middleware versions are distributed through unified base images, keeping environments homogeneous.

⛁

Unified Artifact and Dependency Repositories

Private artifact repositories centrally host dependency packages, locking versions and hashes and preventing environments from pulling different versions independently.

⛓

Dependency Manifests and Lockfiles

Lockfiles solidify the dependency tree so artifacts are built once and reused everywhere, eliminating “same code, different dependencies.”

⛨

Supply Chain Security Scanning

Dependency packages are scanned for vulnerabilities and license compliance, blocking high-risk components from entering the pipeline.

02

Unified Desensitized Test Sample Data

Use centrally governed desensitized sample data to make tests repeatable and comparable while preserving data security and compliance baselines.

☷

Unified Desensitized Dataset

Standardized desensitized samples are derived from production data, centrally maintained, and shared as a common baseline across environments.

⚿

Rule-Based Desensitization

Sensitive fields such as names, IDs, accounts, and amounts are desensitized by unified rules, keeping data usable without leaking secrets.

↻

Repeatable and Versioned

Sample data is version-managed and can reset environment data with one click, ensuring test results are reproducible.

⛒

Tiered Access Control

Different environments obtain data according to sensitivity, while real data in PROD/COB is strictly isolated with least privilege.

Unified CI/CD

One pipeline for unified integration across all environments

From code commits to multi-environment promotion, release, and rollback, everything is orchestrated by unified CI/CD. Environment differences are managed through configuration rather than script forks.

1

Code Commit

Branch strategy + code review + static scanning gates.

2

Unified Build

Build once with unified dependency packages and generate fingerprinted artifacts.

3

Automated Testing

Inject desensitized samples into SIT/UAT and run integration and regression tests.

4

Approval Promotion

Approve based on environment policies and promote artifacts step by step to STAGING.

5

Release and Rollback

Canary release to PROD/COB, with one-click rollback on exceptions.

⚙

Configuration as Environment

The same pipeline differentiates environments through configuration, avoiding separate scripts for each environment.

⛿

Artifact Consistency

Build once and reuse the same artifact throughout the process, fundamentally eliminating differences between environments.

⤺

Observability and Rollback

The release process is fully observable, with health checks enabling fast and safe automatic rollback.
